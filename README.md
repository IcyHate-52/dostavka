# Доставка

Канбан-доска для диспетчера службы доставки.

- **Frontend:** React + TypeScript + Vite + Tailwind CSS + shadcn/ui (Radix primitives)
- **Backend:** Go, plain `net/http`, JSON-file storage behind a `Store` interface

```
dispatch-board/
├── backend/    Go API (см. backend/README.md)
└── frontend/   React app (см. ниже)
```

## Запуск

**1. Backend**
```bash
cd backend
go run ./cmd/server
# -> http://localhost:8080
```

**2. Frontend** (в отдельном терминале)
```bash
cd frontend
npm install
npm run dev
# -> http://localhost:5173
```

По умолчанию фронтенд обращается к `http://localhost:8080`. Чтобы изменить
адрес backend, скопируйте `.env.example` в `.env` и поменяйте `VITE_API_URL`.

## Деплой на Render

Проект собирается в один Docker-образ: Go-сервер раздаёт собранный React
как статику и отвечает на `/api/*` — то есть на Render нужен **один**
сервис, не два.

**1. Загрузите проект на GitHub**
```bash
cd dispatch-board
git init
git add .
git commit -m "init"
```
Создайте репозиторий на github.com и запушьте код туда (через GitHub Desktop
или `git remote add origin ...` + `git push`).

**2. Создайте сервис на Render**
- Зайдите на https://render.com → **New** → **Web Service**
- Подключите ваш GitHub-репозиторий
- Render сам найдёт `Dockerfile` в корне — выберите **Runtime: Docker**
- **Plan:** Free
- Остальные поля можно не трогать — порт Render передаёт сам через переменную `PORT`, сервер уже её слушает

Либо проще: в репозитории уже лежит `render.yaml` — на Render выберите
**New → Blueprint**, укажите репозиторий, и Render создаст сервис по этому
файлу автоматически.

**3. Подождите сборку** (3–5 минут на бесплатном плане) — после неё Render
выдаст адрес вида `https://dostavka.onrender.com`. Открываете его — это
уже и фронтенд, и backend одновременно, на одном домене.

### ⚠️ Важно про бесплатный план Render
- Бесплатные сервисы **засыпают** после ~15 минут без запросов и поднимаются
  заново при следующем заходе (первая загрузка после сна — 30–50 секунд).
- Диск на бесплатном плане **не сохраняется** между перезапусками — после
  "просыпания" или нового деплоя список заказов очистится (файл
  `data/orders.json` создаётся внутри контейнера и не переживает перезапуск).
  Если нужно, чтобы заказы сохранялись постоянно — на платном плане Render
  можно подключить **Persistent Disk** (Render Dashboard → Disks → Add Disk,
  путь монтирования `/app/data`), либо заменить `JSONFileStore` на
  настоящую БД (для этого и сделан интерфейс `Store`).

## Что реализовано (по ТЗ + последние правки)

- Визуальный стиль — в духе крупных сервисов доставки: шрифт Golos Text, жёлтый акцент `#FFCB05` на кнопках, мягкие пастельные плашки-чипы для статусов колонок, скруглённые карточки с мягкой тенью
- Доска **пустая по умолчанию** — никаких тестовых заказов, всё добавляется руками через "+ Создать заказ"
- Кнопка **"+ Создать заказ"** → модальное окно → toast-уведомление "Заказ №… создан!"
- 4 колонки: Новый / Готовится / В пути / Доставлен, со счётчиками
- Карточка заказа: номер, адрес, имя клиента, сумма (зелёным), таймер
- На каждой карточке — кнопки **←** / **→** для смены статуса и кнопка **удаления** (иконка корзины, с подтверждением "Удалить?")
- Заказ в статусе "Доставлен" **автоматически исчезает с доски через 1 минуту** (на backend происходит окончательная архивация чуть позже, фоновой проверкой раз в 15 секунд)
- Drag-and-drop **только на один шаг**; на телефоне drag-and-drop отключён, остаются только кнопки "→"/"←"
- Esc во время перетаскивания отменяет перенос — стандартное поведение браузера для HTML5 drag-and-drop
- Сортировка: "Новый" — старые сверху; остальные колонки — новые сверху
- Просрочка: карточка в "Новом" дольше 30 минут — красная рамка/фон + иконка часов + текст "ПРОСРОЧЕН!"
- Звуки: щелчок при перемещении карточки, колокольчик при статусе "Доставлен" — синтезированы через Web Audio API, без аудиофайлов
- Поиск по адресу с подсветкой/затемнением карточек
- Backend хранит заказы в JSON-файле за интерфейсом `Store`, легко заменить на Postgres/MongoDB

## Одно несоответствие в исходном ТЗ — как я его разрешил

П. 3.0 говорит, что новый заказ появляется **сверху** колонки "Новый", а
п. 3.5 — что в "Новом" сверху должны быть **самые старые** заказы (чтобы их
брали первыми). Это взаимоисключающие правила. Я оставил рабочую логику
диспетчерской (FIFO, старые сверху) как основную, потому что она важнее для
самой работы диспетчера — новый заказ попадёт в самый низ списка "Новый" и
будет обработан по очереди. Если для вас важнее видеть свежесозданные
заказы сразу сверху — скажите, поменяю сортировку.
